# RFC 9000 Section 10.3 — Stateless Reset

## Status: Implemented

## Overview
Stateless Reset allows an endpoint that has lost connection state (e.g., after a restart) to signal the peer that the connection is no longer valid. The mechanism uses a 16-byte token that the peer can recognize without any shared cryptographic state.

## Implementation

### Token Computation (`src/quic/stateless_reset.zig`)
- `computeToken(static_key, conn_id)` — HMAC-SHA256 truncated to 16 bytes
- Deterministic: same key + CID always produces the same token
- Server uses a single `static_reset_key` for all connections

### Packet Generation (`src/quic/stateless_reset.zig`)
- `generatePacket()` — produces a packet that looks like a short header packet
- First byte: long header bit cleared (0x80=0), fixed bit set (0x40=1)
- Random prefix bytes + stateless reset token as last 16 bytes
- Minimum 21 bytes; capped at triggering packet size to avoid amplification

### Detection (`src/quic/stateless_reset.zig`)
- `isStatelessReset()` — checks last 16 bytes against known peer tokens
- Packets < 21 bytes are never stateless resets

### Connection Integration (`src/quic/connection.zig`)
- `static_reset_key: [16]u8` — per-connection key for deterministic token generation
- `LocalCidPool.registerInitialCid()` and `issueNewCid()` accept optional static key
- Server includes `stateless_reset_token` in transport parameters (RFC 9000 §18.2)
- Peer's initial reset token from transport params stored in `peer_cid_pool`
- `matchesStatelessReset(data)` — checks packet against all known peer tokens

### Server-Side Sending (`src/server.zig`)
- Server generates `static_reset_key` at startup
- When a short-header (1-RTT) packet arrives for an unknown CID, sends stateless reset
- Reset packet size capped at triggering packet size (anti-amplification)

### Client-Side Detection (`src/client.zig`)
- On `conn.recv()` error, checks `conn.matchesStatelessReset()`
- If match: logs, sets `conn.state = .draining`, stops processing

## Caveats
- Client MUST NOT include `stateless_reset_token` in transport params (enforced by only setting it for `is_server`)
- The server only sends resets for 1-RTT packets (short header); long header packets for unknown connections are ignored
- Token comparison uses `std.mem.eql` (not constant-time), acceptable since tokens are not secret to the holder
- Server currently supports a single connection; the static reset key would need to be shared across all connections in a multi-connection server
