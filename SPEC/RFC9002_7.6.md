# RFC 9002 Section 7.6 — Persistent Congestion

## Status: Implemented

## Overview
Persistent congestion is detected when ack-eliciting packets sent over a period exceeding the persistent congestion threshold are all declared lost. This indicates the network has been congested for an extended period, triggering an aggressive reduction to minimum congestion window.

## Implementation

### Threshold Calculation (`src/quic/rtt.zig`)
- `persistentCongestionThreshold()` = 3 * (smoothed_rtt + max(4*rtt_var, 1ms) + max_ack_delay)
- Equivalent to 3 * PTO (without exponential backoff)
- Before first RTT measurement: 3 * 2 * 100ms = 600ms

### Detection (`src/quic/ack_handler.zig`)
- `detectLostPackets()` tracks earliest and latest send times of lost ack-eliciting packets
- If `latest_lost_time - earliest_lost_time > persistentCongestionThreshold()`, sets `AckResult.persistent_congestion = true`
- Only ack-eliciting packets count (non-ack-eliciting packets like pure ACKs are excluded)

### Response (`src/quic/congestion.zig`)
- `NewReno.onPersistentCongestion()`:
  - Sets `congestion_window = 2 * max_datagram_size` (minimum window)
  - Sets `ssthresh = congestion_window`
  - Clears recovery state (`largest_sent_at_last_cutback = null`) to allow immediate window growth
  - Resets `bytes_acked_in_round`

### Integration (`src/quic/connection.zig`)
- In ACK processing: if `result.persistent_congestion` is true, calls `cc.onPersistentCongestion()` instead of `cc.onCongestionEvent()`

## Caveats
- Detection happens per packet number space (as each `SentPacketTracker` runs independently)
- The RFC notes that only packets sent after the most recently acknowledged packet should be considered; our implementation considers all lost packets in the current loss detection round, which is slightly more aggressive but safe
