# RFC 9000 Section 8.1.3 — Address Validation for Future Connections (NEW_TOKEN)

## Status: Implemented

## Overview
A server can provide tokens to clients via NEW_TOKEN frames. Clients include these tokens in the Initial packet of future connections, allowing the server to validate the client's address without requiring a Retry exchange.

## Implementation

### Token Format (`src/quic/packet.zig`)
- `generateNewToken(out, client_addr, token_key)` — creates an encrypted token
- Format: nonce(12) + AES-128-GCM(timestamp(8) + addr(14)) + tag(16) = 50 bytes
- Same encryption key as Retry tokens (`token_key` in ConnectionConfig)
- `validateNewToken(token_data, client_addr, token_key)` — decrypts and validates
- Token validity: 60 seconds, address must match

### Server-Side Issuance (`src/quic/connection.zig`)
- After handshake `.complete`, server generates a NEW_TOKEN frame
- Token contains the client's peer address and current timestamp
- Queued via `PendingControlFrame.new_token` (owned buffer, no dangling slices)
- `ConnectionConfig.token_key` supplies the encryption key

### Client-Side Storage (`src/quic/connection.zig`)
- On receiving NEW_TOKEN frame, client stores in `new_token_buf[92]` + `new_token_len`
- `getNewToken()` method returns stored token for application to save
- Application can pass it back via `connect(..., initial_token)`

### Client Reuse (`src/quic/connection.zig`)
- `connect()` accepts optional `initial_token` parameter
- Token stored in `retry_token_buf` and set on `packer.initial_token`
- Included in the Initial packet's token field

### Server Validation (`src/server.zig`)
- When Initial packet has a token, server tries Retry validation first
- If Retry fails, tries NEW_TOKEN validation via `validateNewToken()`
- Valid NEW_TOKEN: connection accepted without Retry (address pre-validated)
- Invalid: packet dropped

## Caveats
- Token validity is 60 seconds (same as Retry tokens)
- Only one token stored per connection (last one wins)
- Application must save and provide the token across connections (not automatic)
- Token key must be the same between server restarts for tokens to remain valid
