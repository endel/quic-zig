# RFC 9000 Section 13.4 — ECN (Explicit Congestion Notification)

## Status: Implemented (protocol-level)

## Overview
ECN allows routers to signal congestion by marking IP packet headers rather than dropping packets. QUIC endpoints report received ECN marks via ACK_ECN frames, enabling the sender to respond to congestion without packet loss.

## Implementation

### ECN Codepoints (`src/quic/connection.zig`)
- `ECN_NOT_ECT` (0b00) — Not ECN-Capable Transport
- `ECN_ECT1` (0b01) — ECN-Capable Transport(1)
- `ECN_ECT0` (0b10) — ECN-Capable Transport(0)
- `ECN_CE` (0b11) — Congestion Experienced
- `RecvInfo.ecn` field carries the IP-level ECN codepoint per received packet

### ECN Counter Tracking (`src/quic/ack_handler.zig`)
- `ReceivedPacketTracker` has `ecn_ect0`, `ecn_ect1`, `ecn_ce` counters
- `onPacketReceived()` increments the appropriate counter based on the `ecn` parameter
- Counters accumulate across the lifetime of each packet number space

### ACK_ECN Generation (`src/quic/ack_handler.zig`)
- `getAckFrame()` returns `Frame.ack_ecn` when any ECN counter is non-zero
- Returns plain `Frame.ack` when all counters are zero
- ACK_ECN frame includes `ecn_ect0`, `ecn_ect1`, `ecn_ce` counts

### ACK_ECN Processing (`src/quic/connection.zig`)
- Full ACK processing (loss detection, RTT, PMTUD, key update tracking)
- ECN validation: detects increase in peer-reported CE count per packet number space
- CE increase triggers `cc.onCongestionEvent()` — same as packet loss response
- Peer ECN counts stored per space: `peer_ecn_ect0[3]`, `peer_ecn_ect1[3]`, `peer_ecn_ce[3]`

### ACK_ECN Frame Codec (`src/quic/frame.zig`)
- Frame type 0x03 (vs 0x02 for plain ACK)
- Parse: reads standard ACK fields + 3 varint ECN counts
- Write: writes standard ACK fields + 3 varint ECN counts

## Caveats
- **No IP-level ECN marking on send**: We do not set ECT(0)/ECT(1) on outgoing IP packets (requires platform-specific `setsockopt` with `IP_TOS`)
- **No IP-level ECN reading on recv**: We do not read ECN bits from received IP packets (requires `IP_RECVTOS` / `recvmsg`); `RecvInfo.ecn` defaults to `ECN_NOT_ECT`
- **Protocol-level only**: All frame parsing, ACK_ECN generation, counter tracking, and CE congestion response are fully implemented. Only the socket-level integration is missing.
- **ECN validation**: Does not implement full RFC 9000 §13.4.2.1 validation (checking that ECN counts are consistent with sent packet counts). Only CE increase detection is implemented.
